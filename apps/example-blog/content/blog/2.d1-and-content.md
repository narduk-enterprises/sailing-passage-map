# D1 and Nuxt Content at the Edge

When you deploy the blog example to Cloudflare Workers, Nuxt Content automatically uses **D1** as the content store. No extra configuration required.

## Why D1?

- **SQLite at the edge** — Same API as local SQLite, runs in every Cloudflare data center.
- **No cold starts** — Content is queried in-process; no network hop to a separate database.
- **Consistent with the rest of the stack** — Auth and other features in this template already use D1 (users, sessions, todos).

## Local vs production

- **Local dev:** Content is read from markdown files (or local SQLite with Nuxt Content’s default).
- **Production (Workers):** Content is stored in D1 and queried at the edge.

Run `pnpm db:migrate:auth` and `pnpm db:seed:auth` in the repo root to seed the shared D1 schema (users, todos) used by the auth and dashboard examples. The blog content itself lives in `content/blog/` and is ingested by Nuxt Content.
